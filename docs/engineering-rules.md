# Huanwrite 工程审计规则

这份文档用于每次接手、重构、评审和合并前的架构审计。

## 审计顺序

1. 读 `AGENTS.md`、`docs/core.md`、`docs/testing-rules.md` 和相关 `spec/`。
2. 确认改动属于哪个模块边界。
3. 确认真相源没有新增第二套。
4. 确认文件职责没有混杂。
5. 确认用户可见能力有真实状态、真实产物和真实验证。
6. 运行 typecheck、单元测试和必要 E2E。
7. 运行坏味道扫描。
8. 同步文档和规格。

## 文件职责

- 一个文件只能有一个主要变化原因。
- 一个文件必须能用一句话说明职责。
- 一个文件也必须能说明不负责什么。
- UI 渲染、业务判断、API 接线、存储读写、配置读取、测试夹具不能混在一个文件里。
- 文件过大不是唯一问题；职责混杂才是问题。
- 文件过碎也不是好架构；同一职责不应被拆成到处跳转。

## 必须集中

- 动作 ID 和动作说明。
- 状态枚举。
- 数据契约。
- 路径规则。
- SQLite 表结构。
- AI provider 配置读取。
- 资产入口。
- 创作稿线定义入口。
- API 契约。

## 必须分离

- 业务服务和路由。
- 业务服务和 UI。
- UI controller hook 和 UI component。
- 图表数据整理和图表渲染。
- 测试夹具和测试断言。
- 存储路径和业务规则。

## 存储规则

- 业务状态使用结构化 SQLite 表。
- 长文本产物使用 Markdown，SQLite 保存路径和关联关系。
- 人类维护资产放在 `.huanwrite/assets`。
- 本机密钥只放 `.huanwrite/.env`。
- 不允许双写同一业务状态。
- 不允许 fallback 读取另一套业务状态。
- 不允许用单表 JSON 文档冒充结构化数据库。

## UI 规则

- UI 只展示状态和触发用户意图。
- UI 不藏业务规则。
- UI 不写假数据。
- 工作台必须能从页面完成真实生产链。
- 后台任务使用 WebSocket 更新。
- E2E 使用可访问名称、业务 ID 和 API 事实定位。

## API 规则

- 路由只做请求校验、调用服务和返回结果。
- 新 API 必须进入 OpenAPI 文档和浏览器 API client。
- 新用户动作必须进入 shared workbench contract。
- API 不返回密钥。
- 错误必须能解释用户下一步该补什么事实。

## 坏味道清单

发现以下问题必须先修：

- 职责混杂。
- 分散真相源。
- 硬编码写作资产。
- 把稿线、平台规则、创作要求写进服务或 UI。
- 空壳按钮、空壳路由、空壳面板、空壳报告。
- 演示图表冒充业务图表。
- `any`、`unknown as`、`as {}`。
- 版本号式功能命名。
- 脆弱 E2E 定位。
- 前端轮询后台任务。
- 构建产物入库。
- 包内测试目录。
- 手写通用基础设施替代成熟库。

## 审计命令

```powershell
npm.cmd run typecheck
npm.cmd run test
npm.cmd run test:e2e
npm.cmd run audit:smells
```

扫描命中必须逐条解释。只有模型名、provider URL 路径或测试自身的坏味道规则文本可以保留。
