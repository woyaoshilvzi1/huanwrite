# Huanwrite UI Target

这份文档记录当前前端目标。它是临时实现目标文档，后续 UI 完成后应并入正式规格。

## 目标

新项目 UI 要做成写稿系统，而不是普通工程看板。用户打开后应立刻理解：我在哪条稿线、当前稿件处于什么状态、下一步该写什么、质量门还差什么、雷达给了什么判断、动作输出在哪里。

目标形态以当前旧写稿工作台为视觉和交互参考：

- 左侧固定窄导航。
- 顶部显示当前页面标题、当前稿线摘要、稿线选择器、刷新、保存。
- 主区域按视图切换：看板、平台雷达、规划、写稿、输出、参考。
- 背景浅灰，卡片白底，边框清楚，绿色作为主操作和选中态。
- 信息密度高，但不要混乱；面向长期写稿使用，不做营销页。

## 视图要求

### 看板

- 顶部指标卡：稿线数、已有正文数、总字数、质量门进度、推进中数量。
- 稿线状态按列展示，不是普通任务卡。
- 每张稿线卡至少显示：标题、稿线类型、当前下一步、质量门进度、字数、正文状态。
- 支持搜索、状态筛选、稿线筛选、拖动改状态。
- 选中稿线要有清晰边框和左侧强调线。

### 平台雷达

- 左侧是动作区：抓取每日雷达、按当前稿线判断、边界说明、快照路径。
- 右侧是可阅读的雷达报告。
- 报告必须能看到：今日编辑结论、今天该做什么、当前稿线匹配、分平台证据评分、噪音词降权、不可推断项。
- 雷达不是装饰面板；雷达信号必须能回到稿线和题材创建。

### 规划

- 顶部显示目标长度、章节数、最近规划、schema 完整、规划填充、待确认、已同步。
- 左侧是章节卡列表，可选中章节。
- 右侧是章节编辑表单。
- 章节编辑必须覆盖：编号、状态、标题、出场角色、目标字数上下限、场景数、当前执行章节、摘要、目标、冲突、变化、转折、钩子、OOC 风险、人工备注。
- 必须有 tabs：章节、人物关系、说话指纹、文风表单、技法卡、模块原文。
- 保存规划、确认规划、刷新规划是正式动作，不是装饰按钮。

### 写稿

- 中间是大正文编辑区，适合长期写作和修改。
- 顶部显示稿线状态、负责人、备注。
- 右侧显示字数目标、链路状态、质量门。
- 链路状态至少覆盖：schema、故事承诺、Beat 表、人物线、关系图、说话指纹、文风指纹、禁用表、场景计划、伏笔账本、写前规划、规划填充、规划确认、规划改动、最新候选、候选 harness、候选审稿、审稿 harness、人工接缝、允许合入。
- 空正文不得覆盖已有正文。

### 输出

- 左侧是运行输出列表，按动作、时间、字数、harness 状态显示。
- 右侧是完整输出正文阅读区。
- 输出不会自动覆盖正文。
- 候选必须经过审稿和合入闸门。

### 参考

- 左侧是参考文件列表。
- 右侧是选中文档正文。
- 参考来源只能来自正式资产、模板、平台规则、规格和当前稿线 story-state。

## 交互原则

- 用户视角优先。不要把内部实现结构直接暴露成页面结构。
- 所有按钮必须连接真实 API、真实状态或真实产物。
- 不要把多稿线误做成普通状态看板。稿线是创作生产线，状态列只是管理视角。
- 不要用大面积空白、薄表单或工程仪表盘冒充写作工作台。
- 不要使用花哨毛玻璃、渐变装饰或营销式英雄区。
- 不要牺牲信息密度；写稿系统应利于扫描、比较、复核和持续操作。

## 实施要求

- React/Vite 保持为前端基础。
- UI 组件只渲染状态和触发意图，不藏业务规则。
- 业务状态来自 API 和 SQLite，不在前端另建真相源。
- 稿线定义来自 `.huanwrite/assets/topic-rules/creative-lanes.json`。
- 新增 UI 必须配真实 API、真实状态和 E2E 验证。
- E2E 必须像真人一样切换视图、选择稿线、编辑规划、保存正文、运行动作、查看输出和核对雷达。

## 验收

完成 UI 实现时必须通过：

```powershell
npm.cmd run typecheck
npm.cmd run test
npm.cmd run test:e2e
npm.cmd run audit:smells
```

如果 UI 与本文档不一致，不能声称完成。
