# Huanwrite 用户偏好

这份文档记录项目所有者的工作偏好。任何新窗口、新 Agent、新维护者都必须先读这里，再动代码。

## 项目目标

Huanwrite 是面向中文网文和短剧生产的本地写稿系统。用户视角优先于技术展示：页面、API、测试和文档都必须服务创作、规划、稿线管理、正文生产、质量门、投稿和雷达决策。

## 沟通偏好

- 使用简体中文回复。
- 直接说事实，不要话术。
- 不要长篇倾倒日志、diff 或源码，除非明确要求。
- 不要把猜测说成事实。
- 不要在没有验证时说完成。
- 不要用技术细节掩盖用户体验问题。
- 不要把“下一步建议”当作完成交付。

## 执行偏好

- 一个明确批次要做完整，再统一验证。
- 不要做半成品、空壳、演示面板或只改局部就声称整体完成。
- 不要把功能缺失包装成取舍。
- 不要在测试没有通过前写完成文档。
- 文档、代码、测试必须同步描述同一个当前事实。
- 除非当前对话明确要求上传或 push，否则禁止 `git push`、发包、发布 release 或任何上传动作。

## 产品偏好

- 前端必须站在写作者视角设计，而不是工程看板视角。
- 多稿线不是普通看板。每条稿线都有平台、受众、长度、题材风向、质量门和创作要求。
- 平台雷达不是展示块。雷达信号必须能进入选题和稿线创作。
- 每条稿线都必须能从雷达或人工选题进入真实创作链路。
- 写作功能必须形成真实闭环：题材、规划、正文、候选、审稿、合入、投稿包、短剧改编、雷达复核。
- UI 要清晰、现代、利于创作，不要毛玻璃、花哨装饰或只有表单的薄页面。

## 架构偏好

- 成熟框架和成熟库能解决的基础能力，用成熟方案；项目代码只做业务胶水和领域逻辑。
- 逻辑是逻辑，UI 是 UI，存储是存储，API 是 API。
- 需要集中的必须集中：状态、动作 ID、路径、配置、契约、资产入口、数据库结构。
- 需要分开的必须分开：渲染、控制器 hook、业务服务、仓库、路由、测试夹具、断言。
- 不要为了集中做上帝文件，也不要为了拆分制造随机碎片。
- 一个文件只承担一个主要职责，并能说明它不负责什么。

## 绝不接受

- 空壳按钮、空壳 API、空壳面板、空壳报告。
- 假测试，只测按钮存在或文案存在。
- 硬编码创作内容、平台规则、质量规则、投稿话术。
- `any`、`unknown as`、`as {}`、假泛型或绕过类型系统的断言。
- `.first()`、`.last()`、顺序猜测、固定像素定位。
- 前端轮询后台任务。
- 构建产物、缓存、临时运行态进入源码管理。
- 双写、历史残留、历史别名、过期接口迁就。
