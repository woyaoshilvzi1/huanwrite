# Huanwrite 用户偏好

这个文件只记录项目所有者不要什么。不要把它写成教程、计划或技术说明。

## 沟通

- 不要用非简体中文沟通。
- 不要长篇大论。
- 不要说空话。
- 不要编造事实。
- 不要把猜测说成事实。
- 不要在没有验证时说完成。
- 不要用术语包装简单事实。
- 不要向项目所有者倾倒大段日志、diff 或源码，除非明确要求。

## 执行

- 不要聊天式推进。
- 不要做半成品。
- 不要留尾巴。
- 不要把功能缺失包装成取舍。
- 不要只改一个局部就声称整体完成。
- 不要改一点测一点再反复口头收尾；先完整实现一个明确批次，再统一验证。
- 不要在测试没有通过前写完成文档。

## 架构

- 不要把逻辑、UI、API、存储、配置和测试夹具混在同一个文件里。
- 不要把集中管理做成上帝文件。
- 不要把职责分离做成随机碎片。
- 不要让 UI 组件藏业务规则。
- 不要让业务服务写页面布局。
- 不要让 API 路由承载业务判断。
- 不要让同一种状态、路径、动作 ID、配置读取或资产入口分散在多个真相源。
- 不要为了“以后可能有用”留下空目录、空 provider、空 registry 或空接口。

## 实现

- 不要写空壳实现。只有按钮、路由、面板、固定报告、演示数据或假结果都不算完成。
- 不要写硬编码写作内容、词库、平台规则、投稿文案或质量规则。
- 不要使用 `any`、`unknown as`、`as {}` 或为绕过类型系统写的断言。
- 不要使用数字后缀式 prompt key 或发布阶段标签冒充当前功能命名。
- 不要在前端用轮询追踪后台任务。
- 不要自己手写成熟框架和成熟库已经解决好的基础能力。
- 不要让构建产物、缓存、临时输出进入源码管理。

## 测试

- 不要用 `.first()`、`.last()`、固定像素、元素顺序或偶然文案猜测目标。
- 不要写只检查按钮存在、不检查真实状态变化的假 E2E。
- 不要把测试放到 `packages/*/tests`。
- 不要用历史黑名单代替当前事实验收。
- 不要只测模型连通性；写作功能必须验证真实 API、真实状态、真实输出和真实 UI 操作。

## 文档

- 不要在正式文档里保留历史过程。
- 不要在正式文档里引用废弃目录。
- 不要写“以后补”“暂时”“先这样”的口径。
- 不要写和当前代码不一致的架构说明。
- 不要把未验证能力写成已完成能力。
